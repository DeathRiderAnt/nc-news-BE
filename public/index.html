NC News API

This API is part of a backend project, designed to interact with data on topics, users, articles and comments.

All endpoints start with /api.

An example would be GET /api/articles, which retrieves a list of articles from the database

GET /api/topics

This will return a slug/title and description for every row in the topics table

GET /api/articles

This will return everything from the articles table, minus the body data, in descending order of creation.

You may query to sort by another field (sort_by) and in ascending or descending order (order).

For example: GET /api/articles?sort_by=author&order=asc

You may also query to return only articles by a certain topic, for example: GET /api/articles?topic=mitch

GET /api/articles/:article_id

This will return all data from an article, within the articles table, that corresponds to whichever article id is requested.

For example: GET /api/articles/3

GET /api/articles/:article_id/comments

This will return all data, minus title and img_url, from all comments that are linked to the requested article id.

For example: GET /api/articles/3/comments

GET /api/users

Thie will retrieve all data from the users table

POST /api/articles/:article_id/comments

This will insert a new comment into the database, linked to whatever article id is requested.

The new comment must be posted in JSON format, with keys of author and body.

For example: POST /api/articles/3/comments

Example JSON to post:

         
            {
            author: 'Barry Chuckle',
            body: 'oh dear, oh dear'
            }

PATCH /api/articles/:article_id

This will update an article by the amount of votes denoted in the request body, incrementing with a positive number or decrementing with a negative.

The request must be a JSON with a key of inc_votes.

For example: PATCH /api/articles/3

Example JSON to patch:

{inc_votes: 5}

DELETE /api/comments/:comment_id

This will delete the comment as requested by comment id.

For example: DELETE /api/comments/2

ERROR HANDLING

Errors that have been tested for should relate to PSQL errors or entries not found in the database, mostly due to input/query error.

Resourses that don't exist should return a status 404 error. Invalid fields should return a status 400 error.

Examples:


            {
            "status": 400,
            "msg": "Bad Request"
            },
            {
            "status": 404,
            "msg": "Not Found"
            }
